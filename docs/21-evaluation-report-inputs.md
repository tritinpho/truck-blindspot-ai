# 21 — Evaluation-Report Inputs (Nội dung 6 / Giai đoạn 6)

The feasibility evidence ([`11-evaluation-plan.md`](11-evaluation-plan.md) §11.7), assembled into a
**report-ready, reproducible** package. For each metric (11 §11.3) and each requirement
([`02-requirements.md`](02-requirements.md)) this maps **target → current evidence → source →
status**, where status is one of:

- **met-in-sim** — the logic/behaviour is demonstrated end-to-end in simulation (L1–L3 + the M3 pipeline).
- **regression-only** — proven as a guard against regressions, but the *headline number* must not be
  quoted from the sim (the simulator derives detection from geometry — it would measure the model
  against itself, [11 §11.2](11-evaluation-plan.md), [14 §P2 #6](14-architecture-critique.md)).
- **NEEDS-L4** — must be measured on the real bench (ESP32 + ultrasonic on the model truck).
- **NEEDS-L5** — must come from the driver/expert usability review.

> **Read this first — the honest headline.** What is proven *now* is that the **logic, the pipeline,
> and the fail-loud safety behaviour are correct and reproducible** (S1–S6 + faults, end-to-end,
> M3). What is **not** a phase-1 deliverable and **must** come from L4 bench / L5 review:
> the **headline detection rate**, the **end-to-end latency**, and the **absolute false-positive
> rate** ([11 §11.2](11-evaluation-plan.md), [14 §P2](14-architecture-critique.md)). Camera/VRU
> classification (FR-11, TC-F3) is **phase-2**, not phase-1 capability. The report must present the
> sim numbers as *indicative / regression* and reserve the headline claims for L4/L5. This document
> keeps that separation in every table.

Companion docs: M3 evidence [`18-m3-summary.md`](18-m3-summary.md); tuning [`19-tuning-and-operating-point.md`](19-tuning-and-operating-point.md);
run book [`17-demo-and-run.md`](17-demo-and-run.md); traceability [`12-traceability.md`](12-traceability.md).

## 21.1 Reproduce every figure (so the report is not anecdotal, 11 §11.6)

All sim figures in §21.4–§21.8 are regenerated by one deterministic command — re-running yields
**byte-identical** numbers (no broker, no wall-clock, no hardware):

```bash
pip install -r tests/requirements.txt

# the whole figure set as Markdown (also writes the canonical logs/events.jsonl):
python tools/eval_report.py
python tools/eval_report.py --out docs/_generated/eval-figures.md   # save a snapshot

# cross-check the canonical replay metric with the independent replay tool:
python tools/log_replay.py logs/events.jsonl

# the operating-point sweep (doc 19) and indicative latency:
python tools/threshold_sweep.py
python tools/scenario_runner.py --latency

# the test evidence itself:
pytest -q tests/ services/fusion-engine/tests        # 105 passed, 2 skipped (broker)
cd apps/hmi && npm test                              # 16 HMI unit tests
```

[`tools/eval_report.py`](../tools/eval_report.py) drives the **real** fusion engine over the scripted
scenarios on a controlled clock; [`tools/log_replay.py`](../tools/log_replay.py) recomputes the
metrics from the emitted `logs/events.jsonl` — and the two agree, so the canonical numbers are
auditable, not typed by hand. **The tables below are a snapshot at the current commit;** the command
regenerates them.

## 21.5 Reproducible-log metrics (canonical run → replay, 11 §11.6)

`tools/eval_report.py` drives **every** scenario into one `logs/events.jsonl`, then
`summarize_events` (== `tools/log_replay.py`) recomputes — snapshot **across the full battery**:

**109 transitions over 8 zones · to-DANGER 9 · to-UNKNOWN 29 · flicker 0.**

| zone | transitions | to-DANGER | to-UNKNOWN | flicker |
|------|:-----------:|:---------:|:----------:|:-------:|
| FRONT | 9 | 0 | 0 | 0 |
| FRONT_LEFT | 16 | 0 | 7 | 0 |
| FRONT_RIGHT | 18 | 1 | 7 | 0 |
| LEFT | 11 | 2 | 0 | 0 |
| REAR | 10 | 1 | 0 | 0 |
| REAR_LEFT | 16 | 0 | 7 | 0 |
| REAR_RIGHT | 16 | 0 | 7 | 0 |
| RIGHT | 13 | 5 | 1 | 0 |

Reading it: **flicker = 0** everywhere (debounce holds — FR-09). The **to-UNKNOWN = 29** are
overwhelmingly the **NFR-12 boot "warming-up"** — the four corner (group-1) zones start UNKNOWN until
their first reading one tick later (7 group-fire scenarios × 4 zones = 28), which is the *correct*
fail-loud behaviour (never a fake SAFE). The single **fault-driven** UNKNOWN is **RIGHT** (TC-F1
unplug). **to-DANGER** concentrates on RIGHT (5) — the highest-`risk_weight`, right-turn-squeeze
zone — exactly the Vietnam right-hand-traffic bias the design weights for.

## 21.6 Indicative danger-path latency (sim, S1–S6)

`tools/eval_report.py` §2 / `tools/scenario_runner.py --latency`. **Indicative only** — the
simulator omits the physical path, so each figure is judged as **est-real = sim debounce + ~70 ms**
(sensor sample+publish ~30 + broker×2 ~10 + HMI render+audio ~30, [03 §3.5](03-architecture.md) / doc 19):

| scn | zone | sim ms | est-real ms |
|-----|------|:------:|:-----------:|
| S1 | FRONT_RIGHT | **100** | **170** |
| S2–S6 | (boosted / static / standby) | n/a | n/a |

Only S1 is a clean *approach* crossing of base `danger_m`; S2–S4 are context-boosted (so DANGER
precedes the base-threshold crossing — warned *early*, not a latency to report), S5 is static-from-start,
S6 is standby. **n=1: p50 100 ms (sim) → ~170 ms est-real, within the 200 ms danger-path budget.**
The **headline** latency is the L4 single-observer measurement
([`tools/latency_observer.py`](../tools/latency_observer.py) on a `--live` run, ADR-0008 #3).

## 21.8 The operating point (NFR-09, tuning trade-off)

Full method + trade-off table: [`19-tuning-and-operating-point.md`](19-tuning-and-operating-point.md).
Summary for the report: a reproducible sweep ([`tools/threshold_sweep.py`](../tools/threshold_sweep.py))
of the debounce levers, scored on **detection latency** vs a **nuisance false-alarm proxy**, judged
against NFR-01 *with the ~70 ms physical path added*. Result: the **designed defaults sit at the knee**
— `confirm=2` (`confirm=3` breaks the latency budget once the physical path is paid; `confirm=1` is
too trigger-happy), `release_margin_m=0.20` (the only `confirm=2` setting with zero flicker). **No
config change was needed; the suite is unchanged.** Caveat carried into the report: nuisance **dwell
is high (~58%)** on a deliberately harsh noise scenario — inherent to a safety-biased hysteresis; the
**absolute** false-positive rate is an **L4** number, and per-zone thresholds stay live-tunable via
`bsw/cmd/set_threshold` so L4 can sweep on the bench without a rebuild.

## 21.10 What L4 (bench) and L5 (usability) must still measure

The report should state these plainly as the path to a full feasibility verdict (11 §11.7):

**L4 — bench (real ESP32 + ultrasonic on the model truck; the W track delivers the node per doc 20):**
1. **Headline detection rate** (≥ 95%) — real objects through real cones, not geometry.
2. **Headline end-to-end latency** — single-observer / SNTP-synced, danger-path ≤ 200 ms (ADR-0008 #3).
3. **Absolute false-positive rate** — real clutter/multipath/ground-bounce; re-run `threshold_sweep`
   with measured bench-noise σ to confirm or nudge the operating point.
4. **Real per-sensor Hz** under group-fire; **recovery time** after a crash; **HW boot** warming-up.

**L5 — usability review (≥ 5 drivers + ≥ 2 experts, [11 §11.5](11-evaluation-plan.md)):**
5. **Glance-time** to name the active zone (≤ 1 s in ≥ 95%); **localization correctness**; Likert
   clarity/trust/perceived-false-alarm — feeding HMI refinements.

**Phase-2 (explicitly not phase-1):** camera/VRU classification (FR-11, TC-F3) and class-dependent
icons/weighting.

Everything else above is **met-in-sim** and reproducible today — M3 de-risked the pipeline so L4/L5
are *measurement* exercises against a system that already works, not open development.
